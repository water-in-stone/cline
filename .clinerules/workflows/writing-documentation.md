# 通用写作指南

# 我希望你如何写作

我要写一些技术性的东西。

通常重要的不是技术细节本身，而是学到新东西或者直接得到一个现成的解决方案。

看，当我阅读时，我想从中获得些什么。所以当我写作时，必须记住读者也想从中获得些什么。这整篇内容的目的就是让任何为我写作或希望我为他们写作的人，了解我对写作产品的看法。

我会列出一个我希望具备的清单，这会让整个写作过程更顺畅。

## 打造引人注目的标题

我经常看到类似"如何用 Y、Z 技术做 X"的标题。这些标题不会让我兴奋，因为 X 或 Y 通常都不是我熟悉的，除非它们本身就很知名。使用 X 很少是梦想本身，除非 X 就是梦想。

我的梦想不是使用 instructor，而是用它提取的数据做些有价值的事情。

一个有效的标题应该：

- 唤起情感反应
- 突出某人的目标
- 提供一个梦想或愿景
- 挑战或评论某种信念
- 解决某人的问题

我认为针对具体问题来写更有影响力。如果这个方法有效，你可以将它复制到各种场景中，而不是保持过于笼统。

- 给所有人的时间管理可以是 15 美元的电子书
- 给高管的时间管理可以是 2000 美元的研讨会

标题要回答你认为每个人都在问的问题，或者解决人们心中有但说不清楚的想法。

不要用"我如何做某事"或"如何做某事"，而是从读者的角度出发用"你如何做某事"来表达。这让标题更有吸引力。但要确保如果内容是主观的，差异是建议性的。"我如何赚了一百万美元"可能比"如何赚一百万美元"更合理，因为你是主体，目的可能是分享你的故事以帮助他人。

这种方法最终训练读者与你的内容产生更强的情感联系。

- "我如何做 X"
- "你如何做 X"

在这两个标题之间，哪个更能引起情感共鸣是显而易见的。

你可以更进一步，添加具体条件。例如，你可以针对特定受众或设定时间框架：

- 如何设置 Braintrust
- 如何在 5 分钟内设置 Braintrust

## 不要用形容词

我希望你几乎总是避免使用形容词，而是用证据代替。不要说"生产就绪"，你可以写"扩展到 100 台服务器或每秒 100 万个文档"之类的内容。这样的数字会准确地告诉你产品的具体表现。如果你不得不用形容词而不是证据，你可能在编造东西。

没有理由说"极速"之类的话，除非这些已经是公认的短语。

相反，说"快 200 倍"或"快 30%"。推荐系统速度提升 30% 已经很惊人了。

200 倍的性能提升是因为我们从一种编程语言换到了另一种。这只是更符合预期和更容易理解的表述。

我最近真正喜欢使用的另一个测试是跟踪你所做的陈述是否能够：

- 被可视化
- 被证伪
- 只有你能说

如果你能同时满足这三点，你的声明更有可能引起受众共鸣，因为只有你能说出来。

今年早些时候，我有一个例子，用 20 美元在 17 分钟内嵌入了整个维基百科，获得了 50 万次浏览。我们只是发布了一个视频，内容是我启动任务，然后你可以看到所有日志行滚过。你看到容器数量从 50 个中的 1 个变成 50 个中的 50 个。

它很容易可视化，可以通过不可复现来证伪。最后，Modal 是唯一能如此轻松做到这一点的公司，这使它独一无二。

## 保持易消化
    - 目标 5 分钟阅读时间
    - 以高中阅读水平写作
    - 分割长段落
    - 使用标题和项目符号

## 使内容可扫描
    - 加粗关键要点
    - 每 3-4 段使用副标题
    - 包含足够的留白
    - 添加相关示例

无论你是写推文串还是完整的博客文章，这种结构都适用。关键是让复杂的想法变得可理解。

# Cline 文档写作指南

## 功能说明的一些通用原则

如果你在讲解一个功能，最好从人性化的解释开始，用简单的语言说明功能是什么。跳过术语，像对从未见过它的人那样解释。这为后续所有内容奠定基础。

将位置和用法合并为一个连贯的部分。告诉用户确切在哪里找到功能以及如何使用它，但将说明融入自然的散文中，在项目符号、编号列表、代码示例（如果适用）、mintlify 组件和标题/副标题之间取得良好平衡。用户不应该在单独的"在哪里"和"怎么用"部分之间来回跳转。

用真实示例展示功能的实际应用，如实际的文件、工作流或代码。用户需要看到具体的实现，而不仅仅是抽象的描述。这是理解转化为实践知识的地方。

谈论功能时，包含一个激发想象力的灵感部分。这个部分通过展示创造性使用该功能时的可能性，将人们从理解推向行动。这就是好文档和优秀文档之间的区别。

## 真正有效的写作原则

### 为行动而写，不仅仅为理解

文档应该激励用户去尝试。与其只是解释某样东西如何工作，不如聚焦于用户能用它实现什么。灵感部分至关重要 - 它是将被动读者转变为主动用户的关键。

### 创建自然的故事流

它应该像一场对话，自然地从"这是什么？"进展到"我如何使用它？"再到"这是一个真实的例子"最后到"想象一下你能用它做什么"。

### 展示真实示例，而非玩具演示

提供实际的工作流文件、真实的代码片段和具体的实现，让用户可以复制和调整。抽象的示例对任何人都没有帮助 - 用户想看到他们实际将要使用的确切内容。

### 保持可扫描但不碎片化

写出阅读完整时自然流畅的散文，但结构化以便用户在排错时能快速找到特定信息。避免密集的文字墙，但也避免过度格式化（过多的项目符号和粗体标题）。应该在所有元素之间有良好的视觉层次平衡，这样你可以快速扫描页面并找到你要找的内容。

## 语言和语调指南

清晰地写作但不要过于浅显。尽可能使用简单的语言，但不要回避用户需要了解的技术术语。用用户能实现什么来解释概念，而不是软件内部如何工作。

让你的写作对话式且鼓励性。像"你也可以试试"或"当那个生效时"这样的短语比僵硬的指令性语言更自然。帮助用户对尝试新事物感到自信。

保持内容简洁且有目的性。每句话要么帮助用户理解某事，要么帮助用户做某事。如果两个目的都不符合，就删掉它。

融入上下文和推理。用户想理解为什么要做某事，而不仅仅是做什么。这能建立信心，并帮助他们在事情不完全按预期工作时进行排错。

## 实践实施

用四部分方法一致地结构化每个功能页面，但让内容在该结构内自然流动。使用视觉资产如视频和截图来补充文字内容 - 它们通常比段落描述更有效地传达信息。

慷慨地链接到相关资源、示例和更深入的文档。用户永远不应该感到卡住或不知道下一步去哪里。维护一个真实示例的仓库，让用户可以参考并适配自己的需求。

目标是让文档更像来自有经验同事的有用指导，而不是技术手册。用户在读完后应该对他们能实现什么感到兴奋，而不仅仅是了解了功能做什么。

## 在结构和灵活性之间取得平衡

虽然讨论了一致的文档结构，但也提到让内容感觉不那么僵硬、更自然。写作应该遵循指南，同时仍然感觉对话式和有吸引力。

# 地道地使用 Mintlify 组件

Mintlify 的自定义组件可以将基础文档转变为引人入胜、可扫描的内容，让用户真正想去阅读。以下是如何有效使用它们。

## 使用 Frame 展示视觉内容

视频和图片应该包裹在 `<Frame>` 组件中，而不是使用原始 HTML 或 markdown。这能创建一致的样式和正确的响应式行为。

对于视频，直接嵌入而不是外链。用户更有可能观看一个 30 秒的演示，而不是点击跳转到另一个平台：

```jsx
<Frame>
	<iframe
		style={{ width: "100%", aspectRatio: "16/9" }}
		src="https://www.youtube.com/embed/your-video-id"
		title="Feature demonstration"
		frameBorder="0"
		allow="accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture; web-share"
		allowFullScreen
	/>
</Frame>
```

截图也类似 - frame 提供视觉上的精致和一致性：

```jsx
<Frame>
	<img src="/path/to/screenshot.png" alt="Descriptive alt text" />
</Frame>
```

## 使用 Card 进行导航和概览

Card 擅长创建可扫描的概览，链接到详细文档。它们非常适合功能列表、入门指南或用户需要选择路径的任何部分。

对相关功能使用两列布局：

```jsx
<Columns cols={2}>
  <Card title="Feature Name" icon="relevant-icon" href="/link/to/docs">
    Brief description that explains what this feature does and why someone would use it.
  </Card>
  
  <Card title="Related Feature" icon="another-icon" href="/another/link">
    Another concise explanation that helps users understand the value proposition.
  </Card>
</Columns>
```

关键是写出足够有信息量的 card 描述，帮助用户决定是否点击进去，但又足够简洁以便快速扫描。每个 card 应该回答"这做什么？"和"我为什么需要它？"

## 使用 Tip 和 Note 提供上下文

使用 `<Tip>` 组件提供有用的信息，增强主内容而不使其杂乱：

```jsx
<Tip>
	Pro tip: You can combine multiple @ mentions in a single message to give Cline 
	comprehensive context about your issue.
</Tip>
```

`<Note>` 组件适合重要的注意事项或技术限制：

```jsx
<Note>
	Due to VS Code limitations, some features require specific settings to work properly.
</Note>
```

`<Info>` 也很好用：

<Info>
	**Quick Fix**: If you're experiencing terminal issues, try switching to a simpler shell like `bash` in the Cline settings.
	This resolves 90% of terminal integration problems.
</Info>

**绝对不要**陷入我们特别指出的那种糟糕的 **粗体文字** - 描述模式。内容应该作为连贯的思想自然流动，而不是感觉像带有强制格式的模板化 AI 回复。


## 何时策略性地使用项目符号和编号列表

项目符号有其功能性用途 - 用于：

**需要用户按特定顺序执行的连续操作或排错步骤**：
1. Install the extension
2. Restart VSCode  
3. Check the settings panel

**用户需要选择一种方法的相关选项列表**：
- Try PowerShell 7 for the most reliable experience
- Switch to Command Prompt if you're still having issues
- Use WSL Bash for Linux compatibility

**用户在解决问题时可能需要快速扫描的快速参考项目**。

**当出现大段文字时改善视觉层次** - 这是引入项目符号或编号列表的好时机。

每个项目符号或编号列表项应该是一个独立的操作或信息，值得在视觉上被分隔开来。这是你追求我之前提到的艺术品体验时可以使用的关键武器。

<good_example_of_bullet_points>
## Finding and Configuring Terminal Settings

You can access Cline's terminal settings by clicking the settings icon in the Cline sidebar, then navigating to the Terminal section. These settings control how Cline interacts with your system's terminal.

- The **Default Terminal Profile** setting determines which shell Cline uses for executing commands. If you're experiencing issues, this is usually the first thing to change. I personally keep this set to `bash` on all my systems because it's the most reliable option, even though I use `zsh` for my regular terminal work.

- **Shell Integration Timeout** controls how long Cline waits for the terminal to become ready. The default is 4 seconds, but if you have a heavy shell configuration (lots of plugins, slow startup scripts), you might need to increase this to 10 or even 15 seconds. I've found that WSL environments and SSH connections often need longer timeouts.

- The **Enable Aggressive Terminal Reuse** setting determines whether Cline tries to reuse existing terminals even when they're not in the correct directory. When this causes problems (commands running in the wrong directory, virtual environment issues), disabling it creates more terminal instances but ensures each command runs in a clean state.

- **Terminal Output Line Limit** sets how many lines of output Cline will read from commands. The default of 500 lines works for most cases, but you might need to adjust it. For verbose build outputs, increase it. For commands with progress bars that spam thousands of lines, decrease it to around 100 to avoid consuming too many tokens.
</good_example_of_bullet_points>

## 像人类一样写作，而不是 AI

保持简短和直接。如果你能用更少的词说清楚，就这样做。冗长的解释往往比它们帮助的更让人困惑。

使用正常的句子长度。自然地混合短句和长句，就像你跟人说话时一样。避免那些无限延伸的复合句。

去掉官腔。不要用"utilize"，说"use"。不要说"in order to"，直接说"to"。写得像在向同事解释东西，而不是在起草新闻稿。

不要过度解释显而易见的事情。如果用户在读文档，他们可能已经理解基本概念了。尊重他们的智商。

## 绝对不要使用破折号或表情符号

就这样。绝对不要用。只有 AI 才用破折号和表情符号写作。

# 拟人化 Cline

提到 Cline 时，始终称他为"him"而不是"it"。

反面示例：
- When Cline can't execute commands or read their output, you lose access to one of its most powerful capabilities.

正面示例：

- When Cline can't execute commands or read their output, you lose access to one of his most powerful capabilities.

# 分享你的工作流时使用"I"

加入个人化的感觉很有帮助。目前文档中有很好的例子，我用"I"来分享我个人如何使用 cline，开发者对开发者。这是一种很好的技巧。

# 交叉链接相关文档页面

写完文档后确保添加交叉链接。如果有相关文档，就链接到它们。

# 简洁是智慧的灵魂

如果不需要就不要啰嗦。使用项目符号和编号列表。保持内容易于阅读。
